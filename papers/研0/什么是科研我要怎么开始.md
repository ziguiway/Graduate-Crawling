## 一、浅谈科研入门

### 1.1 从论文选题到发表的完整流程

```
研究方向 → 具体问题 → 技术创新 → 实验评估 → 论文写作
```

### 1.2 如何选择研究方向

**途径：**
- 问导师、师兄师姐
- 自己多读论文、多交流
- 读顶会会议论文
- 多听报告、多参加会议
- 交流访问（有一定科研基础后再出去）
- 了解科研动态和领域牛人

**好的研究方向应具备的条件：**
- 要有实际需求（重要）
- 能吸引人（新颖）
- 有较大的未知空间（非显而易见）
- 实验室有好的相关积累（基础）

### 1.3 如何选择具体研究问题

一般来说，解决一个具体的研究问题，就能发一篇Paper。

**确定具体研究问题的方法：**
- 找导师要Idea
- 自己读论文，从研究前沿推敲可以做的题目

**具体研究问题的两种类型：**

| 类型 | 特点 | 研究者贡献侧重 |
|------|------|----------------|
| **新问题** | 因新需求、新技术产生（如元宇宙、大数据、实时分析） | 清晰地定义一个好的新问题 |
| **老问题** | 已被前人定义好（如人机对弈围棋、轨迹相似度查询） | 新技术 |

**新问题需要注意：**
- 需要明确定义这个问题，为什么重要
- 需要有较好的技术贡献
- 要有完整的故事链条和恰当的技术贡献

**老问题的做法：**
- 更快、更高、更强（提高性能、准确率、通用性）
- 充足的实验比较
- 应用到新的环境、满足新需求（新硬件、新场景、隐私要求、算力不足等）

### 1.4 如何判断一个研究问题的价值

**新问题：**
- 目标：解决没人研究过的问题
- 条件：有真实的应用背景、能获得真实的数据集、结果好衡量

**老问题：**
- 目标：更好的方法
- 问题已经明确定义

### 1.5 如何找论文

- **90%的精力**集中在领域的顶会顶刊
- **10%的精力**放在交叉领域
- 常用数据库：SIGMOD、VLDB、ICDE、KDD、TKDE、VLDB Journal、IEEE VIS、TVCG、CHI等
- 参考 CCF A类列表或清华推荐的顶会顶刊列表
- 常用文献调研网站：谷歌学术、Connected Papers、DBLP

### 1.6 如何读论文

**找方向时：**
- 广度优先，看idea，不注重细节

**找到感兴趣方向时：**
- 深度优先，看细节、实验、数据集
- 能够实现别人代码
- 抓住论文要害，批判式阅读

## 二、做研究需要哪些素质

### 2.1 如何做研究

- **脚踏实地（努力）**：不能熊瞎子掰棒子，掰一个丢一个
- **规划能力（远见）**：了解相关工作，做好实验（知彼）；认清论文质量（知己）
- **多总结**，多借助于工具，总结经验教训

### 2.2 做研究的大忌

- 没有兴趣
- 不了解前人的工作
- 浮躁、急于求成、不求甚解
- 马虎
- 懒惰

### 2.3 经验分享

- 找准科研方向、沟通交流很重要
- 撸起袖子去干、快速迭代不拖延
- 脚踏实地、写好每一行代码
- 仰望星空、打开想象的空间
- **三四月做的事情，八九月自有答案**
- 一定要保证科研时间

### 2.4 建立和谐的师生关系

- 目标一致
- 相互欣赏
- 共同进步
- 契约精神

## 三、学术论文撰写

### 3.1 如何衡量一个工作的价值

- 问题新颖程度
- 方法创新程度
- 技术理论深度
- 实验结果好坏（与现有工作对比）
- 论文撰写功底（逻辑结构、英语）

### 3.2 论文结构（八股文）

```
摘要 → 引言 → 相关工作 → 整体框架 → 技术细节（3个创新点）→ 实验 → 总结
```

### 3.3 摘要写作（What - Why - How - So What）

- **What**：研究什么问题？交代背景
- **Why**：为什么写这篇论文？提出新问题/发现新规律，或旧问题新方法（更快、更好、成本节约）
- **How**：大概怎么做？Key Idea
- **So What**：实验效果如何？

### 3.4 引言写作

- 摘要的扩充版，论文的总结版
- 包含：研究动机（讲故事）、问题背景/应用前景、基本Idea、解决什么问题、创新点、文章结构
- **吸引审稿人，几乎决定文章能否被录取**
- 注意事项：突出创新点、突出技术深度/新颖性、明确说出和现有方法的区别

### 3.5 文章结构比英语更重要

- 结构清晰：总分结构、Leading text、Highlight
- 贯穿全文的形象化例子（尤其是复杂理论或方法）
- 图文并茂、言简意赅
- 避免重复文本，尽量少用复合句
- **目标：站在读者的角度，把故事讲清楚、讲好**

### 3.6 解决方案概览（Overview / Framework）

- 总分架构，每章都是总分结构
- 第一段概述本章结构，凝练整个论文贡献
- 介绍整个workflow和architecture
- 概述每个技术点/module的用途和目的
- **要图文并茂，通过图来解释架构**
- 读者读了这章基本了解论文做法
- 一定要自包含，要有leading text

### 3.7 "技术创新"部分的写作

- 不要指望读者能很容易理解你的技术点
- **把读者当"傻子"**，先介绍简单的，循序渐进展开
- 合理使用图、表和公式
- 尽量用例子来解释难懂的技术部分
- 逻辑一定要清晰连贯
- 如果有算法，最好提供伪代码并进行解释
- 复杂的算法可以增加例子来阐述

### 3.8 相关工作（Related Work）

- 引用相关论文，说出具体区别
- **多夸奖别人、承认别人的贡献**（审稿人可能就是该文作者）
- 保证语言正确，不要曲解相关工作的真实意思

### 3.9 结论

- 总结文章贡献
- 不要与引言重复
- 和引言语调不同（引言：读者还不知道细节；结论：读者已看完论文）

### 3.10 如何取标题

- 用一句话概括所做的工作
- 模板：**问题 + 方法 + 贡献**
- 可以适当"别出心裁"

### 3.11 论文撰写建议

1. **组织全文的叙事逻辑**（禁忌想到什么写什么）
2. **初稿严禁直接使用ChatGPT生成！！！**（强调多次）
3. 组织故事线（全文结构）
4. 确定每个部分写什么内容
5. 不同部分之间的逻辑联系和起承转合
6. 写完一段之后请读一遍（发现低级错误）
7. 多使用简单句，把事情讲清楚
8. 注意用好连接词
9. 用ChatGPT、Grammarly**检查语法**（而非生成内容）

### 3.12 常见问题

- 第三人称单数、冠词、一句话不能有两个动词、不要用复杂句子
- 用词要地道合理，不要随便查字典就放上去
- 符号简洁清晰，定义了就要用
- **画图要用心、美观，一个图想两天、画一天很正常**
- 两句话之间加不上连词的时候就要小心了
- 逻辑还是第一位的，千万不要让人去猜逻辑、猜内容，be specific

### 3.13 排版建议

- 用对相关学术论文的投稿模板
- 尽量学习使用LaTeX进行论文排版
- 在线协作进行LaTeX编辑（Overleaf）
- **一定要重视画图！画图！画图！**

### 3.14 审稿人的角度

**希望看到：**
1. Novel Problem：新问题，尤其是定义一个有用的新问题
2. Novel Method：解决旧/新问题的新颖有效的方法
3. Nice Story：好的写作、吸引人的故事、逻辑性强
4. Nice Presentation：排版美观、插图漂亮（避免低级语法/排版/拼写错误）

**讨厌看到：**
1. Old school problem, with simple combination of existing methods
2. Worse Presentation：writing, figures
3. Experiment：without strong baselines, inappropriate settings

### 3.15 论文投稿与被拒

- 伤心一两天就让他过去吧
- **不要气馁**：任何人的论文都有可能被拒
- 顶会顶刊接受率一般只有20%左右
- 找原因，补短板，适当根据Reviewer的意见修改打磨

## 核心要点速查表

| 维度 | 关键建议 |
|:----:|:--------:|
| **选题** | 新老问题皆可，关键是要有真实应用背景和数据 |
| **读论文** | 广度优先找方向，深度优先做研究 |
| **做研究** | 脚踏实地、快速迭代、不拖延 |
| **写论文** | 结构比英语重要，逻辑清晰、图文并茂 |
| **画图** | 一个图想两天画一天很正常 |
| **心态** | 被拒正常，持续改进 |
| **师生关系** | 目标一致、相互欣赏、共同进步、契约精神 |
